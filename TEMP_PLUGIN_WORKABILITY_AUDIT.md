## Карта плагина

- `core`: жизненный цикл, модули, команды, tab-complete, тексты, PlaceholderAPI.
- `config`: `BalanceConfig`, загрузка `config.yml`, `quests.yml`, `pets/*.yml`.
- `player`: `PlayerDataManager`, SQLite/JSON storage, dirty tracking, статистика, квестовый прогресс.
- `egg`: ядро питомца, offhand-кнопка, синхронизация инвентаря, смерть/респавн, summon charge.
- `pet`: runtime-питомец, движение, бой, социальное поведение, эволюция, GUI-facing state.
- `pet/ability`: пассивные эффекты, автолут, защитная реакция, utility-события.
- `pet/inventory`: PetVault, хранение вещей питомца.
- `gui`: главное меню, питомец, помощь, квесты, кузница редкости, NPC binding.
- `quest` + `economy`: задания, очки, антиабуз, лутбокс-попытки.
- `box`: базовые коробки и физические алтари лутбоксов.
- `master`: встроенный Пет-Мастер.
- `api`: публичные API для питомцев, прогресса и экономики.

## P1 - сначала чинить

На этом проходе явных P1 по статическому аудиту не осталось. Следующие риски уже ниже по приоритету или требуют серверного runtime-теста.

## P2 - после P1

1. Active item PDC хранит не весь `OwnedPetData.progress`.
   - Сейчас часть состояния живёт только в PlayerData: `defense`, `follow_position`, `first_summon_done`, appearance keys.
   - Для обычной игры нормально, но перенос item-only может терять настройки.

## P3 - позже, когда основные ветки живые

1. `/vpc reload` перезапускает CoreModule через `disable()`/`enable()`, но не все сервисы являются модулями.
   - `NpcGuiBindingService`, `PetGuiService`, API и часть runtime state не имеют отдельной reload-логики.

2. `ModuleManager` не изолирует ошибки модулей.
   - Один exception в `enableAll()`/`reloadAll()` может остановить последующие ветки.

3. `PetVaultService` читает и пишет YAML на main thread при автолуте.
   - На маленьком сервере терпимо, но при большом item spam может проседать.

4. `quests.yml.generator-pools` сейчас не используется кодом.
   - Это либо задел на генератор, либо мёртвый конфиг.

5. `messages/ru.yml` и `messages/en.yml` почти не участвуют в выводе.
   - Большинство текстов захардкожены в `GameText` и command/gui коде.

6. Документы `CODEX_CONTEXT.md`, `MODULE_TREE.md`, `TEST_CHECKLIST.md` выглядят как старые mojibake-артефакты.
   - На runtime не влияет, но мешает ориентироваться.

## План работ

1. Укрепить загрузку пользовательских YAML:
   - `npc-gui.yml` уже защищён по UUID;
   - `pet-master.yml` сейчас мягко игнорирует пустой world;
   - остальные ручные YAML стоит проверить на тот же стиль защиты.

2. Решить, нужно ли сохранять полный `OwnedPetData.progress` в PDC яйца.

3. Затем пройти reload/storage/performance:
   - `ModuleManager` guard;
   - reload-aware сервисы;
   - PetVault IO.

4. После этого отдельно проверить горячие игровые циклы:
   - `PetEngineManager.updateAll()` и `grantActivityXp()`;
   - `PetInteractionEffects` и похожие визуальные эффекты;
   - entity churn вокруг мастера/лутбоксов на долгих сессиях.

## Что уже выглядит нормально

- `plugin.yml` команды и permissions в целом совпадают с command handler.
- `pets/*.yml` есть для всех 17 `PetType`.
- `PlayerData` dirty tracking покрывает списки, карты, статистику и nested pet progress.
- SQLite/JSON storage компилируется и имеет fallback.
- Quest/economy базово связаны: accept, turnin, reward, extra box attempt.
- Lootbox altar, Pet-Master и NPC GUI имеют рабочую основную механику, кроме устойчивости к ручной порче файлов.
