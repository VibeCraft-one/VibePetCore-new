# VibePetCore Context

## Что это
`VibePetCore` — Paper/Purpur-плагин про питомцев-спутников.
Основа: ядро питомца в `offhand`, призыв по `ПКМ`, runtime-пет рядом с игроком, кормление, эволюция, боевые и utility-реакции, GUI, квесты и очки.

## Текущая механика
- Источник правды по активному питомцу: `PlayerData.activePetId` + `OwnedPetData`.
- Простое перекладывание ядра во вторую руку не призывает пета само по себе.
- Призыв: ядро во вторую руку -> `ПКМ` -> отсчёт -> спавн.
- Первый призыв отдельный: 4 секунды, золотые искры перед игроком, потом появление.
- Если питомец активен и его ядро во второй руке, предмет меняется на маленькую зачарованную кнопку.
- Если ядро убрано из второй руки, runtime-пет скрывается, но данные игрока остаются целыми.

## Визуальные договорённости
- Над питомцем: коротко, например `Лиса [Э2 Ур.4]`.
- Action bar: коротко, например `[HP 18/20 Еда 4/5]`.
- Лор яйца:
  - без устаревающего таймера;
  - конкретные бонусы игроку;
  - штраф за смерть показывается только пока реально активен.
- Ранние формы должны выглядеть малышами; рост по эволюциям заметный.

## Нюансы и слабые места
- Самая хрупкая зона: синхронизация `PlayerData <-> предмет <-> runtime pet`.
- Проверять сценарии:
  `join`, `respawn`, `world change`, `creative/survival`, перекладывание предмета, рестарт сервера.
- Старые яйца прошлых версий могут вести себя странно; для зачистки есть `DANGER-DELETE`.
- Коты и волки теперь запоминают внешний вид через `OwnedPetData.progress()`.

## Админка
- `/vpc DANGER-DELETE All`
- `/vpc DANGER-DELETE <player>`
- После ввода админ должен подтвердить действие словом `yes` в течение 10 секунд.
- `All` поднимает `egg-core.purge-epoch`, чтобы старые яйца прошлых поколений стали невалидны.

## Repo hygiene
- `src/main/resources/config.yml` должен хранить только placeholder-значения для MySQL; реальные DB-данные живут только в серверном локальном конфиге.
- DB-пароль, который уже попадал в GitHub до 2026-06-21, считать скомпрометированным и заменить на сервере.

## GUI refactor
- Текущий безопасный курс: одна самостоятельная GUI-страница = отдельный `*Page` класс.
- `PetGuiService` пока остаётся listener/facade, чтобы не ломать команды, события и cooldown guards.
- Уже вынесены: `SourceMainPage`, `SourceQuestPage`, `SourceBoxPage`, `SourceForgePage`, `SourceLegendaryPage`, `SourceHelpPage`, `PetArmorHelpPage`, `GrowthMissingPage`, `PetInfoPage`, `PetOverviewPage`.
- `PetOverviewPage` содержит render, click-handler и item-builder'ы личного `/pet` экрана; `PetGuiService` держит действия/синхронизацию.
- `SourceQuestPage` содержит render/click-handler квестов Источника, категории и пагинацию; `quests:*` маршрутизируется через `PetGuiRouter`.
- `PetGuiRouter` и `GuiPageId` должны постепенно забирать новые страницы и клики.
- Не переносить все страницы одним большим патчем; после каждой страницы запускать `compileJava test`.
